# 单分支请求、AI 处理状态与聚合采样 — 详细研发方案

## 1. 范围与目标

- **单分支**：每次预处理请求仅接受一个 `branch`，多分支通过多次请求 + 单项目单任务串行。
- **状态持久化**：在 PostgreSQL 记录预处理状态，用于并发控制（同一 project_id 仅允许一个 running）与前端查询。
- **聚合采样**：类/包/模块/项目聚合时，每种节点类型仅取前 3 个下级参与 LLM 输入，且提示词中必须注明「此为采样」。

---

## 2. API 规范（详细）

### 2.1 预处理触发

- **方法与路径**：`POST /api/projects/{project_id}/preprocess`
- **请求体**（JSON）或 **Query**：
  - `branch`：string，必填或默认 `"main"`，本次只处理该分支。
  - `force`：boolean，可选，默认 `false`；为 true 时忽略已有 description/hash 强制重算。
- **成功**：HTTP 200，body 示例 `{ "status": "running", "project_id": 1, "branch": "main", "message": "任务已启动" }`（或 202 Accepted，任务异步执行）。
- **冲突**：HTTP 409 Conflict，body 示例 `{ "detail": "项目正在处理中，请稍后重试", "code": "PROJECT_BUSY" }`。当该 `project_id` 下已存在 `status='running'` 时返回。
- **其他错误**：404（project 不存在）、500（服务异常）。

### 2.2 状态查询

- **方法与路径**：`GET /api/projects/{project_id}/preprocess/status`
- **Query**：
  - `branch`：string，可选；不传则返回该项目下所有 branch 的最近状态列表；传则返回该 project+branch 的最近一条记录。
- **成功**：HTTP 200，body 示例：
  - 单 branch：`{ "project_id": 1, "branch": "main", "status": "completed", "started_at": "...", "finished_at": "...", "extra": { "saved": 100, "skipped": 20 } }`
  - 多 branch：`{ "items": [ { "project_id": 1, "branch": "main", "status": "completed", ... }, ... ] }`

## 3. 数据库设计（实现级）

### 3.1 迁移文件

- **路径**：`docker/migrations/006_ai_preprocess_status.sql`

### 3.2 表结构

```sql
CREATE TABLE IF NOT EXISTS ai_preprocess_status (
    id          SERIAL PRIMARY KEY,
    project_id  INT         NOT NULL,
    branch      VARCHAR(255) NOT NULL,
    status      VARCHAR(32)  NOT NULL,
    started_at  TIMESTAMPTZ NOT NULL,
    finished_at TIMESTAMPTZ NULL,
    error_message TEXT NULL,
    extra       JSONB NULL,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (project_id, branch)
);

CREATE INDEX IF NOT EXISTS idx_ai_preprocess_status_project_running
    ON ai_preprocess_status (project_id) WHERE status = 'running';

COMMENT ON TABLE ai_preprocess_status IS 'AI 预处理任务状态，用于单项目单任务并发控制与状态查询';
COMMENT ON COLUMN ai_preprocess_status.status IS 'pending|running|completed|failed';
COMMENT ON COLUMN ai_preprocess_status.extra IS '可选统计，如 saved_count, skipped_count, layer_count';
```

- **status 枚举**：`pending`、`running`、`completed`、`failed`。
- **唯一约束**：`(project_id, branch)` 唯一，每次任务结束对该 branch 做 upsert 覆盖；若需保留历史可后续改为「每次任务一行」+ `task_id`。

### 3.3 并发规则（实现逻辑）

- **允许启动新任务**：当前不存在 `project_id = $project_id AND status = 'running'` 的记录。
- **占位**：插入或更新 `(project_id, branch)` 为 `status='running'`, `started_at=now()`, `updated_at=now()`；若上面查询已存在 running，则**不**写入，直接返回 409。
- **结束**：将该 `(project_id, branch)` 更新为 `status='completed'|'failed'`, `finished_at=now()`, `error_message`（失败时）, `extra`（可选）, `updated_at=now()`。

## 4. 服务层实现要点

### 4.1 状态仓储

- **位置**：`src/service/repositories/ai_preprocess_status_repository.py`（或并入现有 repository）。
- **接口**：
  - `has_running(conn, project_id: int) -> bool`：存在该 project_id 且 status='running' 则 True。
  - `set_running(conn, project_id: int, branch: str) -> bool`：若 has_running 为 True 则返回 False；否则 upsert (project_id, branch) 为 running，返回 True。
  - `set_completed(conn, project_id: int, branch: str, extra: dict | None)`：更新为 completed、finished_at、extra。
  - `set_failed(conn, project_id: int, branch: str, error_message: str)`：更新为 failed、finished_at、error_message。
  - `get_status(conn, project_id: int, branch: str | None) -> list[dict]`：按 project_id 或 (project_id, branch) 查询最近记录。

### 4.2 预处理入口与并发控制

- **位置**：`src/service/services/ai_preprocessor_service.py`（或 `ai_preprocessor.py` 中对外入口）。
- **流程**：

  1. 解析 `project_id`、`branch`（默认 main）、`force`。
  2. 获取 DB 连接，`has_running(conn, project_id)`；若 True，返回 409。
  3. `set_running(conn, project_id, branch)`（若上步为 False，此处应成功）。
  4. 在 try 中调用 `Preprocessor.run(project_id, branch, force)`，收集返回的统计（如 saved, skipped）。
  5. finally：成功则 `set_completed(conn, project_id, branch, extra={...})`，异常则 `set_failed(conn, project_id, branch, str(e))`。
  6. 返回 200 或 202 及适当 body。

### 4.3 路由挂载

- **位置**：`src/service/routers/` 下新建 `preprocess.py` 或并入 `projects.py`。
- **POST /api/projects/{project_id}/preprocess**：调用 4.2 的 service，根据返回值设置 HTTP 状态码与 body。
- **GET /api/projects/{project_id}/preprocess/status**：调用状态仓储 `get_status`，返回 JSON。

## 5. 聚合采样与提示词（实现级）

### 5.1 适用节点类型

- 对以下**每种**聚合层级均采用「下级采样 + 提示词注明采样」：**Class**（下级为 Method/Function）、**Package**（若存在，下级为 Class 等）、**Module**（若存在）、**Project**（若存在）。

### 5.2 采样规则

- **配置**：`aggregation_sample_size`，默认 **3**（每父节点最多取 3 个下级）。
- **顺序**：下级列表按 `name` 升序（或按 `id` 升序）排序后，取前 `aggregation_sample_size` 条；无 name 时用 id。
- **可选**：单条描述最大长度 `aggregation_max_desc_chars`（如 200），超长则截断并加「…」；或整体拼接后 `aggregation_max_chars` 截断。

### 5.3 提示词规范（必须包含的说明）

- 在聚合 prompt 的**正文中**（非仅注释）必须包含一句明确说明为采样的中文，例如：
  - 「以下为下级节点的**采样**（共 {total} 个，此处仅展示前 {n} 个作为代表），请基于该采样推断本节点的整体职责与作用。」
- 变量：`total` = 下级总数，`n` = 实际参与本轮的个数（min(3, total)）。
- 实现时在 Preprocessor 内封装一个 `_build_aggregate_prompt(scope, name, items, total)`，其中 `items` 为已采样的 (name, desc) 列表，`total` 为原始下级总数；在 prompt 模板中写死上述说明或使用占位符。

### 5.4 Preprocessor 内实现位置与步骤

- **位置**：`src/service/ai/preprocessor.py` 或 `src/service/services/ai_preprocessor.py` 中「聚合类/包/模块/项目描述」的循环内。
- **步骤**：

  1. 获取当前父节点的下级列表 `children`（名称 + 描述）。
  2. `total = len(children)`；对 `children` 按 name（或 id）排序，取前 `aggregation_sample_size` 条得到 `sampled`。
  3. 用 `sampled` 拼成「名称: 描述」的文本块，再调用 `_build_aggregate_prompt(scope, name, sampled, total)`，其中 prompt 内包含「此为采样，共 total 个，此处仅展示前 n 个作为代表」。
  4. 调用 LLM 得到父节点描述，再向量化并写回 Neo4j。

## 6. 配置项汇总

| 配置键 | 类型 | 默认值 | 说明 |

|--------|------|--------|------|

| aggregation_sample_size | int | 3 | 每种节点类型聚合时下级采样数量 |

| aggregation_max_desc_chars | int | 可选 | 单条下级描述最大字符数，超则截断 |

| aggregation_max_chars | int | 可选 | 整段「下级描述」文本最大字符数，超则尾部截断 |

可从环境变量或 [config.example.json](NeoDev/src/config.example.json) 读取，与现有 sync/AI 配置方式一致。

---

## 7. 实施顺序建议

1. **迁移**：编写并执行 `006_ai_preprocess_status.sql`。
2. **状态仓储**：实现 `ai_preprocess_status_repository` 的 has_running / set_running / set_completed / set_failed / get_status。
3. **预处理入口**：在 service 层封装「占位 → run → 收尾」流程，与 Preprocessor.run 对接，并依赖状态仓储做 409 判断。
4. **路由**：注册 POST preprocess 与 GET preprocess/status，返回码与 body 符合第 2 节。
5. **Preprocessor 聚合**：在各类/包/模块/项目聚合处增加「排序 → 取前 N → 拼 prompt 时带采样说明」逻辑，并接入 `aggregation_sample_size` 配置。

---

## 8. 与主计划的衔接

- **主计划第 7 节（预处理）**：参数仅单 `branch`；聚合阶段按本方案第 5 节实现采样与提示词。
- **主计划第 9 节（API）**：以本方案第 2 节为最终 API 规范；409 表示项目正在处理中。
- **主计划第 4 节或新增**：PG 侧增加 `ai_preprocess_status` 表及状态枚举，逻辑以本方案第 3、4 节为准。

按本研发方案实现后，即满足：每次请求仅一个分支、AI 处理状态持久化与单项目单任务并发控制、每种节点聚合时统一采样前 3 个下级且在提示词中注明采样。
