# 数据结构设计文档（影响面分析）

> 基于 [影响面分析页面 PRD](.cursor/plans/影响面分析页面_prd_ef560444.plan.md)，描述 **现有** 与 **规划** 的 PostgreSQL、Neo4j 数据结构，以及 PG 与 Neo4j 的关联方式。  
> 环境基线见 [docker-compose.yml](../docker-compose.yml)。  
> 本文档在表结构与图模型设计上 **为后续领域驱动设计（DDD）预留扩展能力**：边界清晰、实体可映射为聚合根/实体，便于按限界上下文拆分与演进。

---

## 一、概述

| 存储 | 现状 | 规划/扩展 |
|------|------|-----------|
| **PostgreSQL** | 仅启用 pgvector 扩展，无业务表；后端 repos/parse 未持久化到 PG | 新增项目/版本/需求/提交/影响面分析等业务表，可选扫描状态表 |
| **Neo4j** | 已有代码图：节点 `(id, branch)` 唯一，单分支标量 | 多分支模型：节点 `branches: list[str]`，按分支过滤子图 |

## 二、PostgreSQL

### 2.1 现有结构（当前代码/基础设施）

- **库**：`neodev`（[docker-compose.yml](../docker-compose.yml)）。
- **初始化脚本**：[docker/init-pgvector.sql](../docker/init-pgvector.sql) 仅创建扩展：

```sql
CREATE EXTENSION IF NOT EXISTS vector;
```

- **业务表**：当前 **无**。仓库解析、解析任务、分支列表等均未持久化到 PG（解析结果写入 Neo4j，扫描状态在仓库本地 `.gitnexus/scan_state.json`）。

### 2.2 规划表结构（PRD 3.2）

以下为 PRD 规定的表草案，供实现阶段建表与 API 使用。

#### 2.2.1 projects（项目）

| 列名 | 类型 | 说明 |
|------|------|------|
| id | PK, 自增或 UUID | 主键 |
| name | VARCHAR | 项目名称 |
| repo_path 或 repo_url | VARCHAR/TEXT | 本地路径或远程仓库 URL |
| created_at | TIMESTAMPTZ | 创建时间 |
| （可选）watch_enabled | BOOLEAN | 是否启用 Git 监听 |
| （可选）neo4j_database / neo4j_identifier | VARCHAR | 多 Neo4j 实例时标识，用于与 Neo4j 图关联 |

#### 2.2.2 versions（版本/分支）

版本与分支 **1:1**。

| 列名 | 类型 | 说明 |
|------|------|------|
| id | PK | 主键 |
| project_id | FK → projects.id | 所属项目 |
| branch | VARCHAR | Git 分支名，唯一约束 (project_id, branch) |
| version_name 或 tag | VARCHAR | 可读版本名或 tag，可选 |
| created_at | TIMESTAMPTZ | 创建时间 |
| （可选）last_parsed_commit | VARCHAR(40) | 该分支上次增量扫描的 commit，供监听与 pipeline 使用 |

#### 2.2.3 requirements（需求）

| 列名 | 类型 | 说明 |
|------|------|------|
| id | PK | 主键 |
| project_id | FK → projects.id | 所属项目 |
| title | VARCHAR | 标题 |
| description | TEXT | 描述，可选 |
| external_id | VARCHAR | 外部系统 ID，可选 |
| created_at | TIMESTAMPTZ | 创建时间 |

#### 2.2.4 commits（提交）

| 列名 | 类型 | 说明 |
|------|------|------|
| id | PK | 主键 |
| project_id | FK → projects.id | 所属项目 |
| version_id 或 branch | FK/VARCHAR | 所属版本/分支 |
| commit_sha | VARCHAR(40) | 提交 SHA |
| message | TEXT | 提交信息 |
| author | VARCHAR | 作者 |
| committed_at | TIMESTAMPTZ | 提交时间 |

建议对 (project_id, commit_sha) 或 (project_id, version_id, commit_sha) 做唯一约束，避免重复同步。

#### 2.2.5 requirement_commits（需求–提交 多对多）

| 列名 | 类型 | 说明 |
|------|------|------|
| requirement_id | FK → requirements.id | 需求 |
| commit_id | FK → commits.id | 提交 |

主键建议 (requirement_id, commit_id)。

#### 2.2.6 impact_analyses（影响面分析任务/结果）

| 列名 | 类型 | 说明 |
|------|------|------|
| id | PK | 主键 |
| project_id | FK → projects.id | 所属项目 |
| status | VARCHAR | 状态，如 pending / running / completed / failed |
| triggered_at | TIMESTAMPTZ | 触发时间 |
| result_summary 或 result_store_path | TEXT/VARCHAR | 结果摘要或存储路径，可选 |

#### 2.2.7 impact_analysis_commits（分析–提交 多对一）

多笔提交对应一次分析。

| 列名 | 类型 | 说明 |
|------|------|------|
| impact_analysis_id | FK → impact_analyses.id | 分析任务 |
| commit_id | FK → commits.id | 提交 |

#### 2.2.8 监听/扫描状态（二选一或组合）

- **方案 A**：在 **projects** 上增加 `watch_enabled`；在 **versions** 上增加 `last_parsed_commit`（如上）。
- **方案 B**：单独表 **branch_scan_state**，例如：

| 列名 | 类型 | 说明 |
|------|------|------|
| project_id | FK | 项目 |
| branch | VARCHAR | 分支名 |
| last_parsed_commit | VARCHAR(40) | 该分支上次扫描到的 commit |

唯一约束 (project_id, branch)。监听服务与 pipeline 据此判断增量/全量及 `since_commit`。

## 三、Neo4j

### 3.1 现有图模型（当前实现）

代码与约束定义见：[neo4j_writer.py](../src/gitnexus_parser/neo4j_writer.py)、[graph/types.py](../src/gitnexus_parser/graph/types.py)。

#### 3.1.1 Neo4j 节点类型（Node Labels）

以下为 **图模型内所有节点类型**（与 [graph/types.py](../src/gitnexus_parser/graph/types.py) 中 `NodeLabel` 一致）。**有唯一约束** 的 Label 用于 MERGE 与索引，当前仅对部分核心结构/代码元素建约束（见 3.1.2）。

| 节点类型 (Label) | 说明 | 当前是否建约束 |
|------------------|------|----------------|
| **Project** | 项目（图内） | 是 |
| **Package** | 包 | 是 |
| **Module** | 模块 | 是 |
| **Folder** | 目录 | 是 |
| **File** | 文件 | 是 |
| **Class** | 类 | 是 |
| **Function** | 函数 | 是 |
| **Method** | 方法 | 是 |
| **Variable** | 变量 | 是 |
| **Interface** | 接口 | 是 |
| **Enum** | 枚举 | 是 |
| **Struct** | 结构体 | 是 |
| **Namespace** | 命名空间 | 是 |
| **Trait** | Trait（Rust 等） | 是 |
| **Impl** | Impl 块 | 是 |
| **Community** | 社区（图分析） | 是 |
| **Process** | 过程（图分析） | 是 |
| **Decorator** | 装饰器 | 否 |
| **Import** | 导入（可作节点） | 否 |
| **Type** | 类型 | 否 |
| **CodeElement** | 通用代码元素 | 否 |
| **Macro** | 宏 | 否 |
| **Typedef** | 类型定义 | 否 |
| **Union** | 联合体 | 否 |
| **TypeAlias** | 类型别名 | 否 |
| **Const** | 常量 | 否 |
| **Static** | 静态成员 | 否 |
| **Property** | 属性 | 否 |
| **Record** | 记录 | 否 |
| **Delegate** | 委托 | 否 |
| **Annotation** | 注解 | 否 |
| **Constructor** | 构造函数 | 否 |
| **Template** | 模板 | 否 |

- **扩展**：新增节点类型时在 `NodeLabel` 与（若需 MERGE 唯一性）`CONSTRAINT_LABELS` 中增加，即可在 DDD 的“代码图”限界上下文中作为新实体类型使用。

#### 3.1.2 约束（当前）

对 **CONSTRAINT_LABELS** 中的 Label 创建 **节点唯一约束**（存在则跳过）：

- 约束形式：`(n.id, n.branch) IS NODE KEY` 或 `(n.id, n.branch) IS UNIQUE`。
- 当前列表：`File`, `Folder`, `Project`, `Package`, `Module`, `Class`, `Function`, `Method`, `Variable`, `Interface`, `Enum`, `Struct`, `Namespace`, `Trait`, `Impl`, `Community`, `Process`。

即：**当前为单分支维度，每个节点 (id, branch) 唯一**。

#### 3.1.3 节点（当前）

- **主键维度**：`id`（字符串）、`branch`（字符串，单值）。
- **写入**：`MERGE (n:Label {id: $id, branch: $branch}) SET n += $props`。
- **通用属性**（来自 [types.py](../src/gitnexus_parser/graph/types.py) NodeProperties，按需存在）：

| 属性 | 类型 | 说明 |
|------|------|------|
| name | string | 名称 |
| filePath | string | 文件路径 |
| content | string | 内容，可选 |
| startLine / endLine | int | 行范围 |
| language | string | 语言 |
| isExported | bool | 是否导出 |
| description | string | 描述 |
| heuristicLabel | string | 启发式标签 |
| cohesion | float | 内聚度 |
| symbolCount | int | 符号数量 |
| keywords | list[string] | 关键词 |
| enrichedBy | "heuristic" \| "llm" |  enrichment 来源 |
| processType | "intra_community" \| "cross_community" | 过程类型 |
| stepCount | int | 步骤数 |
| communities | list[string] | 社区 |
| entryPointId / terminalId | string | 入口/终止节点 |
| entryPointScore / entryPointReason | float/string | 入口评分/原因 |

#### 3.1.4 关系类型（Relationship Types）

来自 [types.py](../src/gitnexus_parser/graph/types.py) 的 `RelationshipType`，用于 DDD 中“代码图”限界上下文内关系建模与影响面遍历。

| 关系类型 | 说明 | 典型端点 |
|----------|------|----------|
| **CONTAINS** | 结构包含 | Folder/File 层级 |
| **CALLS** | 调用 | Function/Method → Function/Method |
| **IMPORTS** | 导入 | File → File |
| **INHERITS** | 继承 | Class → Class |
| **OVERRIDES** | 重写 | Method → Method |
| **USES** | 使用 | 符号引用 |
| **DEFINES** | 定义 | 容器 → 成员 |
| **DECORATES** | 装饰 | Decorator → 被装饰节点 |
| **IMPLEMENTS** | 实现 | Class → Interface |
| **EXTENDS** | 扩展 | Class → Class |
| **MEMBER_OF** | 归属 | 成员 → 容器 |
| **STEP_IN_PROCESS** | 过程步骤 | Process 内步骤 |

关系写入时通过端点 `(id, branch)` 匹配；关系属性：`id`, `confidence`, `reason`（及部分类型的 `step` 等）。

#### 3.1.5 扫描状态（当前，非 Neo4j）

- 存于 **仓库本地**：`<repo_path>/.gitnexus/scan_state.json`。
- 格式：`{ "branch_name": "last_scanned_commit_sha", ... }`。
- 读写见 [incremental.py](../src/gitnexus_parser/ingestion/incremental.py)（`load_scan_state` / `save_scan_state`）。

### 3.2 规划：多分支模型（PRD 2.4 / 3.3）

- **节点**：由单值 `branch` 改为 **`branches: list[str]`**，同一节点可属于多个分支。
- **唯一性**：按 **id**（及 Label）唯一，不再用 (id, branch) 作为节点键；具体约束形式以多分支改造计划为准。
- **查询某分支子图**：`WHERE $branch IN n.branches`。
- **“变更数据”**：新分支 B 从已扫描分支 A 的某提交签出时，不复制子图，仅对原属 A 的节点执行 `SET n.branches = n.branches + $newBranch`（或等价 `add_branch_to_nodes(driver, source_branch=A, new_branch=B)`）。
- **删除某分支**：从节点的 `branches` 中移除该分支；若 `branches` 为空则删节点。
- 关系端点仍按 **id** 匹配（多分支下同一 id 唯一节点）。

## 四、PG 与 Neo4j 的关联

### 4.1 关联方式（PRD 3.3）

- **项目**：PG 的 **project** 对应一个代码仓库；可在 **projects** 表或配置中记录 `neo4j_database` / 实例标识（多 Neo4j 时）。
- **分支**：PG 的 **version.branch**（或 **branch_scan_state.branch**）对应 Neo4j 中“该分支”的子图。
- **查询子图**：
  - **当前**：`WHERE n.branch = $branch`。
  - **多分支改造后**：`WHERE $branch IN n.branches`。
- **定位**：用 `(project_id, branch)` 在 Neo4j 中唯一定位该仓库该分支的子图（若多项目共用一 Neo4j，则图内需能通过 project 或 branch 命名约定区分）。

### 4.2 影响面分析（未实现，PRD TODO）

- 输入：PG 中选中的 **commit_ids**。
- 目标逻辑（待实现）：根据提交得到变更文件/符号，在 Neo4j 中基于 **分支过滤** 查询 CALLS/IMPORTS/CONTAINS 等做影响面计算，结果写回 PG（如 impact_analyses.result_summary / result_store_path）。
- 本文档仅定义表与关联，不涉及该逻辑实现。

## 五、实体关系小结

```
PG:
  projects 1──N versions (version.branch 与分支 1:1)
  projects 1──N requirements
  projects 1──N commits
  versions 1──N commits (commits 归属某 version/branch)
  requirements N──M commits (requirement_commits)
  impact_analyses N──1 commits (多提交 → 一次分析，impact_analysis_commits)

PG ↔ Neo4j:
  (project_id, version.branch) → Neo4j 子图过滤条件 (branch 或 branches 含该分支)
```

## 六、为 DDD 预留的扩展能力

本小节说明当前数据结构如何支持后续 **领域驱动设计（DDD）**：限界上下文划分、聚合边界、实体与值对象映射，以及扩展方式。

### 6.1 限界上下文与存储映射

| 限界上下文（建议） | 主存储 | 核心实体/聚合根（示例） | 扩展方向 |
|--------------------|--------|----------------------------------|----------|
| **项目管理** | PG | Project, Version, Requirement, Commit | 后续可增加 Bug、TestSuite 等表 |
| **影响面分析** | PG + Neo4j | ImpactAnalysis（PG）；图遍历基于 Neo4j 子图 | 结果明细、策略配置可落 PG 或 Neo4j |
| **代码图 / 解析** | Neo4j | 节点类型见 3.1.1，关系类型见 3.1.4 | 新增 Node Label / Relationship Type 即新概念 |
| **Git 监听与扫描** | PG（+ 本地 JSON） | branch_scan_state 或 versions.last_parsed_commit | 与“项目管理”共享 Version，与“代码图”共享分支维度 |

- **PG 表**：每张业务表对应可识别的 **实体**；主表 + 关联表可视为一个 **聚合**（如 Project 聚合、ImpactAnalysis 聚合）。新增领域时 **新增表或新列**，避免在单表上堆砌所有概念。
- **Neo4j**：整图视为“代码图”限界上下文；**节点 Label = 实体类型**，**关系 Type = 领域关系**。新增节点类型或关系类型即扩展该上下文；其他上下文（如 Bug、测试）若需引用图节点，可通过 **节点 id + project_id/branch** 在 PG 中做外键或冗余存储。

### 6.2 聚合与一致性边界（建议）

- **Project 聚合**：projects + versions（及可选 branch_scan_state），以 project_id 为边界；Version 创建/删除与分支监听状态变更在同一事务或 saga 内考虑。
- **Requirement 聚合**：requirements + requirement_commits，以 requirement_id 为边界；提交的绑定/解绑不直接改变 Commit 本身。
- **ImpactAnalysis 聚合**：impact_analyses + impact_analysis_commits；一次分析对应多提交，结果写回 impact_analyses，可扩展 result 明细表。
- **代码图**：Neo4j 内以 **分支子图** 为一致性单元（当前单 branch，规划中 branches 数组）；与 PG 无强事务，通过 (project_id, branch) 做最终一致性关联。

### 6.3 扩展方式小结

- **PG**：新领域（如 Bug、测试）→ 新表 + 与 projects/commits/requirements 的 FK；新属性 → 新列或 JSONB/扩展表，避免单表爆炸。
- **Neo4j**：新节点概念 → 在 `NodeLabel` 与（若需）`CONSTRAINT_LABELS` 中增加 Label；新关系概念 → 在 `RelationshipType` 中增加类型；多分支改造后节点唯一键为 id（+ Label），关系仍按端点 id 匹配。
- **跨上下文引用**：PG 实体引用图节点时，存 **node_id（字符串）+ 所属 project_id/branch**，便于在 Neo4j 中定位；图侧不直接存 PG 主键，由查询时通过 (project_id, branch) 限定子图再按 node_id 查找。

按上述方式演进，现有表结构与图模型可直接作为 DDD 中的 **持久化模型** 与 **领域模型** 的映射基础，无需大改即可引入应用服务层、领域层与仓储抽象。
